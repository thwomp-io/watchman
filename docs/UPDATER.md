# Self-update — how Watchman updates itself in place

Watchman's bundled installs (Windows NSIS especially) self-update via
[tauri-plugin-updater](https://v2.tauri.app/plugin/updater/): the app checks the project's GitHub
releases for a `latest.json` manifest, downloads the new installer, verifies its **minisign
signature** against the public key baked into the app, applies it, and relaunches. Install once,
update from the footer — no full reinstall.

## How the pieces fit

| Piece | Where | Job |
|---|---|---|
| `tauri.conf.json → plugins.updater` | base config | **Points nowhere** (empty endpoints, placeholder pubkey) — dev/local builds never self-update |
| `tauri.release.conf.json` | overlay config | Adds `createUpdaterArtifacts` + the GitHub `releases/latest/download/latest.json` endpoint; CI merges it over the base for signed release builds |
| `#[cfg(not(debug_assertions))]` | `src-tauri/src/lib.rs` | The updater/process plugins only register in release builds — a debug binary can't even ask |
| Capabilities | `capabilities/default.json` | Exactly three grants: `updater:allow-check`, `updater:allow-download-and-install`, `process:allow-restart` |
| Footer UpdatePill | `src/UpdatePill.tsx` / `src/update.ts` | Check → show version/notes → download w/ progress → "restart to update"; offline check-failure degrades to a quiet inline note |
| CI bundle job | `.github/workflows/ci.yml` | When the signing secret exists: signed Windows build + `.sig` + assembled `latest.json`, all attached to the release |

The signing keypair here is the **updater's own trust root** (minisign-style, generated by the
Tauri CLI). It is independent of OS code-signing (Authenticode / notarization) — the unknown-publisher
question is a separate decision; this key is only what stops a tampered download from installing.

## One-time key setup (the operator, in your own terminal)

The private key must never transit chat, the repo, or CI logs — generate it yourself:

```sh
cd bus-app
npx tauri signer generate -w ~/.tauri/watchman-updater.key
```

This prints the **public key** and writes the **private key** (you'll be prompted for a password).

1. **Bake the public key into the app**: replace `TAURI_UPDATER_PUBKEY_PLACEHOLDER` in
   `bus-app/src-tauri/tauri.conf.json` (`plugins.updater.pubkey`) with the printed public key.
   The public key is not a secret — committing it is correct.
2. **Store the private key + password as GitHub Actions secrets** on the public repo:
   - `TAURI_SIGNING_PRIVATE_KEY` — the *contents* of `~/.tauri/watchman-updater.key`
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — its password
3. **Back the key up** somewhere durable and offline. Losing it means shipping a new public key —
   i.e. one final manual reinstall for every user before self-update works again.

Until the secrets exist, CI's signed lane is skipped and the plain unsigned build runs — nothing
breaks; there's just no self-update channel yet.

## What the FIRST real release after enabling must verify

- [ ] `latest.json` + the versioned `*-setup.exe` + its `.sig` all appear as release assets, and
      `https://github.com/<org>/<repo>/releases/latest/download/latest.json` resolves.
- [ ] A previously-installed build (with the real pubkey baked in) shows the update in the footer,
      downloads, verifies, applies, and relaunches at the new version.
- [ ] **The bundled Python engine survives the update.** The engine payload is a Tauri *resource*
      (`bundle.resources: {"engine": "engine"}`), so it lives inside the bundle the updater
      replaces — an update ships the new engine with it. Verify a post-update `engine doctor` /
      first dashboard load works.
- [ ] **The Windows first-run bug class does not recur on the UPDATE path** — the installer-context
      uv-junction failure ("untrusted mount point", os error 448) was a *fresh-install* bug; the
      update path re-runs the NSIS installer and must be regression-tested on a real Windows
      machine. This cannot be verified headlessly — a human confirms on a real Windows machine.
- [ ] The updater's version comparison respects the release's SemVer tag (the app version comes
      from `CARGO_PKG_VERSION`, kept in lockstep across the three version files).

## Version-channel notes

- **Releases are the only update channel.** Dev/local builds point at no endpoint; the updater
  never phones anywhere from a dev build.
- A release's SemVer tag is the source of truth for what version a release *is*; the updater
  simply follows whatever `latest.json` the release CI attached.
- macOS: the updater requires a signed app; local builds rely on a locally-provisioned self-signed
  identity (Tauri falls back to ad-hoc signing when none is configured). CI does not build macOS
  bundles (unsigned CI artifacts would be Gatekeeper-quarantined), so macOS self-update stays
  local-build territory until that changes.
